
general guidelines: 
- keey branch names short and simple 
- use only lowercase. 
- the type of branch is generally the same with the type of the issue you are working on. 

main 
- this is the main branch
- there must be no other branch with the name main/something 

dev 
- this is the dev branch.
- similar to main branch, there must be no other branch with the name dev/something

release
- start with release followed by a forward slash and the version of the release. 
- e.g. release/v1.0.0


hotfix 
- for fixing issues in the main branch
- start with hotfix followed by a forward slash and the name of issue and dash and the ticket number of the issue. 
- e.g. hotfix/issue-name-#100 


feature 
- start with feat followed by a forward slash and the name of the feature and a dash and the ticket number of the issue. 
- e.g. feat/feature-name-#100
- feature names must follow the kebab case naming convention 

enhance 
- start with enhance followed by a forward slash and the name of the feature and a dash and the ticket number of the issue. 
- e.g. enhance/feature-name-#100 
- feature names must follow the kebab case naming convention 

bug 
- for fixing issue in the dev branch
- start with bug followed by a forward slash and the name of the bug and a dash and the ticket number of the issue 
- e.g. bug/button-not-working-#65 
- bug names must follow the kebab case naming convention 

refactor 
- start with refactor followed by a forward slash and name of what you are refactoring and a dash and the ticket number of the issue
- e.g. refactor/feature-name-#100
- follow the kebab case naming convention

docs
- start with docs followed by a forward slash and name of what you are documenting and a dash and the ticket number of the issue
- e.g. docs/feature-name-#101
- follow the kebab case naming convention
